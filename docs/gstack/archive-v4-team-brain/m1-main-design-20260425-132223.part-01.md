# Design: insights-share — repo-scoped precedent router for Claude Code sessions

Generated by /office-hours on 2026-04-25
Branch: main
Repo: V4teamBrain (~/projects/V4teamBrain) + insights-share plugin (~/projects/insights-share)
Status: APPROVED
Mode: Startup

## Problem Statement

A 4-engineer team (founder + 3 teammates) repeatedly hits the same Claude Code session pitfalls. Lessons one teammate's session learns do not propagate to other teammates' sessions. The most-Claude-Code-fluent engineer (founder) becomes a human broadcaster, group-syncing lessons in person — costing his time and arriving too late for teammates already burning hours in the same loops. The plugin's purpose is to share lessons between Claude Code sessions WITHOUT humans being the carrier.

The thesis goes beyond "humans-as-broadcaster fails." It claims **humans-as-learner also fails**: any human-targeted lesson channel (wiki, standup, "how-to-use Claude Code" blog) is a leaky pipe. AI sessions are the only scalable carrier of organizational learning. Existing AI memory tools (claude-mem, Memorix, Cowork enterprise plugins) all assume "single source of truth / git truth merging," which collides with the reality that the same problem in squad A and squad B has different correct solutions. insights-share rejects truth-merging in favor of preserving competing context.

## Demand Evidence

- **Yesterday's incident (founder direct observation)**: a teammate spent 8 hours stuck in an `ai-quick-demo new-feature → bug → fix → bug → fix` loop. Founder had previously learned the workaround ("docs first, do not constrain how Claude implements, let it run new ai-quick-demo-with-new-features end-to-end") but had not communicated it. The lesson would have saved the full 8 hours.
- **Founder's recurring sync cost**: founder repeatedly group-calls teammates to broadcast lessons learned. Founder identifies it as the actual bottleneck (not the lessons themselves).
- **No paying users. No external users.** 4 internal users.

Single-anecdote-per-week density of repeated-loop incidents. Sufficient to justify a wedge prototype; insufficient to justify skipping wedge validation (founder has accepted this risk by choosing Approach B).

## Status Quo

Q2 answer: **A — nothing is transmitted, every teammate re-steps the same trap**. No Slack-broadcast workaround, no shared `lessons.md`, no `CLAUDE.md` lesson file, no team wiki. Each Claude Code session is a fresh slate. Duct-taped human channel (founder calling group syncs) is the only mitigation.

## Target User & Narrowest Wedge

**Target user (immediate)**: the 4 V4team engineers. Founder is producer-consumer; 3 teammates are receiver-only at minimum, producer-consumer if zero-touch capture works.

**Founder's stated wedge (Q4 verbatim)**:
> "install this plugin and just download and use skills in user unaware status, upload new learned lessions in user unaware status, viral spread between claude code instances through collaboration projects."

Three actions, all user-unaware: install + auto-use; upload learned; viral spread within same git-remote.

**Ontology rule (Q4 push)**: no `solution` tags. Only `good example` / `bad example` + raw context (jsonl + commit id + git link). Agent decides which example fits asking session's context.

**Codex steelman**: **"insights-share is a repo-scoped precedent router."** Sessions leave good/bad examples + raw context; later sessions retrieve relevant prior failures before repeating.

## Constraints

- Plugin already exists with 71/71 tests + 28/28 UAT + 6 canned answers + statusline + force-install marker + marketplace setup. ~70-80% of current scope is scaffolding around the wedge, not the wedge itself.
- 4 internal users. No external distribution required for v1.
- Same-git-remote is the privacy boundary. **Canonicalization rule**: same `<host>/<owner>/<repo>` regardless of HTTPS/SSH protocol; computed via `git remote get-url origin` then normalized (strip `.git` suffix, lowercase host).
- v1 ships only to **trusted-org repos** (read-access ≈ write-access ≈ lesson-visibility). Per-folder ACL deferred to v2.
- **Definition of "user-unaware"**: no UI prompts, no opt-in dialog during capture/upload, no observable latency > 500ms in any Claude Code interaction. The only user-aware moment is the initial `claude plugin install` consent. Once installed, the plugin is silent until a teammate runs `/insight-rate` or `/insight-flush`.

## Premises

**P1 — pain real**: team repeats the same Claude Code session lesson. Yesterday's 8h incident is canonical evidence.

**P2 — anti-human channel (broadcaster)**: AI sessions transmit lessons directly between each other, bypassing human broadcasters.

**P3 — anti single-truth ontology**: no `solution` tag for lesson content. Only `good example` / `bad example` body labels + raw context. Agent decides which example fits asking session's context. Explicitly rejects Memorix-style git-truth merging. **Routing metadata (topic tag) is separate from this rule** — topic is a retrieval key, not a solution authority. **Codex independently flagged P3 as the sentence that defines the product.**

**P4 — carrier = git remote (trusted-org scope v1)**: viral spread limited to same canonical git-remote. Alice and Bob in same repo → both auto-install → mutual lesson visibility. Cross-org isolation by construction. **v1 explicit decision**: read-access on remote ≈ lesson-visibility (no per-folder ACL). Single-remote scoping per session; install isolates per remote. Per-folder ACL deferred to v2 post-wedge. **Trade-off accepted**: contractor / intern in same repo sees senior lessons. Mitigation: do not deploy to mixed-trust repos until v2.

**P5 — zero-touch + filter pipeline (defended against Codex challenge)**: capture and upload are user-unaware. Founder's defense: **users do not know what they did is a lesson**. A web-designer who solves a backend problem via "simple, fast debug" never tags it as a lesson because to them it is normal. Any user-aware capture systematically misses the most valuable subset (unconscious lessons). PII solved at filter pipeline:
  1. **Layer 1**: hard-coded append-only regex filter (API keys, common secret patterns, $HOME paths, /private paths, JWT-shape tokens, AWS/GCP key shapes)
  2. **Layer 2**: haiku/claudefast agent single-pass (privacy + topic inference together)
     - Input schema: `{turn_text: string, prior_redactions: span[]}`
     - Output schema: `{redact_spans: [{start, end, reason}], confidence: 0-1, topic_tags: string[]}`
     - Single haiku call returns both redaction spans AND topic tags (cost-amortized; topics are routing metadata only, not solution authority — see P3)
     - Timeout: 2s per turn; on timeout → **drop turn from upload** (turn still seen by user in their own local session; only the upload to mirror is suppressed)
     - Fallback: if agent returns confidence < 0.8 → **drop turn from upload** (same semantics)

**P5 hard gate (Gate 0)**: before any raw jsonl flows live, build gold-corpus benchmark — **≥500 real jsonl turns from founder's own `~/.claude/projects/` history, manually labeled for PII**. Pipeline must achieve **0 PII leaks + ≤2% false-positive rate** on this corpus before live capture is enabled. **Corpus diversity required**: ≥3 different project types (web / infra / data, or other distinct domains the team works on) AND ≥50 deliberate-injection adversarial patterns founder seeds (known API key shapes, customer names, internal acronyms) to verify recall — this turns Gate 0 from retrospective into adversarial. Failure to pass Gate 0 = B is unbuildable as designed and must revise. **Founder labeling effort estimate: 8-16 hours (one-shot).**

**P6 — current plugin is mostly scaffolding**: 71 tests / 28 UAT / canned answers / statusline / force-install / marketplace are real craft but mostly NOT the core wedge. Core wedge (raw-jsonl-router + viral via git remote + filter pipeline) is not yet built.

**P7 (epigram)**: **"humans are hard to learn; Claude Code sessions self-learn is the ground-truth."** All human-targeted lesson channels are leaky pipes. AI session is the only scalable carrier of organizational learning. Strengthens P2 from "humans fail as broadcasters" to "humans fail as learners too."

## Approaches Considered

### Approach A: Minimal Viable Wedge (Codex 48h prototype + filter pipeline)
- Stop hook silently captures lesson + UserPromptSubmit hook injects top-3 prior examples, scoped to same git remote
- Two-layer filter (hard-coded regex + haiku privacy agent)
- Seed 10-20 historical sessions, 4-engineer 2-day run, hard metrics
- **Effort**: M (1-2 weeks). **Risk**: Low-Med
- Pros: validates wedge with hard metric; same-git-remote + 2-layer filter satisfies P4 P5 + Codex challenge; retains thesis; 1-2 weeks vs 2-3 months
- Cons: forces immediate decision on what to do with 70% scaffolding (delete vs maintain)
- Reuses: existing hooks, `/insight-add` repurposed, marketplace as V4team install path

### Approach B: Ideal Architecture (full thesis at once)
- Full raw-jsonl-router + central per-repo `<owner>/<repo>-insights` mirror repo (plain committed jsonl) + retrieval via hooks (no MCP server in v1) + lazy install via first UserPromptSubmit
- **Build effort**: 8-12 weeks (≈ XL "2-3 months"). **Time-to-Gate-1**: ~4 months from start (build + 4-week B run). **Sunk cost if Gate 1 fails**: 6-9 months including reverse-migration off plugin and team workflow restoration. **Risk**: High
- Pros: thesis fully realized; defensible architecture; category-defining potential if thesis correct
- Cons: skips wedge validation; routing latency unmeasured pre-build; haiku/storage cost real (see Cost Model); unobserved-usage risk (Q5: founder has not watched teammates use plugin)
- Reuses: existing plugin shell, hook framework

### Approach C: Lesson Harvester (no jsonl)
- LLM-extract lessons from commit messages, PR descriptions, post-mortems, `# WARN`/`# TODO` comments, RFC.md
- **Effort**: S (1 week). **Risk**: Low
- Pros: zero PII risk; team workflow unchanged; immediately OSS-able
- Cons: **kills thesis core** — commit messages are coarse signal, miss metacognitive content of raw chat. Capture-blindness layer (P5 web-designer example) unreachable.
- Reuses: git, PR data, plugin shell

## Approach Decision (Recommended A — Rejected; Founder Chose B)

**A — Recommended (rejected by founder)**. Recommended because: Q5 surprise data is empty; yesterday's 8h evidence is 1 anecdote, not 4-user diagnostic; v1 already over-built — another 2-3 months pre-validation increases sunk cost dramatically.

**B — Chosen (founder bet)**. Founder's verbatim reasoning: "this is a common case observed within orgs. but i have NEVER even noticed such discussions. what i see is people loving running after new claude code how-to-use share instead of realizing (1) humans are hard to learn (2) claude code sessions self learn is the ground-truth."

Recorded as a **founder bet, not a derived decision**. If thesis correct, B is right. If thesis wrong, A would have caught it in 2-3 weeks; B costs 6-9 months including reverse-migration.

**Gate 1 (4-week run after B build) is this bet's kill switch — abandon B if Gate 1 fails. Do not iterate. Do not extend the run. Reverse migration begins week 17 if Gate 1 fails.**

### B-Scope Locked Decisions (no longer open)

Lesson metadata in mirror's root `lessons.jsonl`. **No git LFS** — jsonl is small (4MB/day team-wide est) and text-compressible. Single-file upper bound = 50MB; split if exceeded (well under GitHub's 100MB hard limit). |

Stored at `~/.gstack/insights-auth/<repo-slug>.token` (chmod 600). Onboarding: `/insight-install` prompts for PAT once per repo, validates scope (`<repo>-insights` write), persists. v2: migrate to GitHub App for org-wide install. |
| **Cold-start (mirror discovery)** | First UserPromptSubmit hook in the repo lazy-detects missing mirror clone at `~/.gstack/insights/<repo-slug>/`. If mirror clone missing → background-clone (up to 5s wait); if mirror does not exist on remote (no commits yet) → log warning + disable retrieval until manually created (`gh repo create <owner>/<repo>-insights`). **Retrieval reads from local cache only — no network in hot path.** Background sync runs every 30 min via OS scheduler / Claude Code idle hook. |
| **Stop hook semantics** | Capture trigger = Claude Code **Stop hook** (fires at end of each agent response, NOT per tool call NOR per turn). Append response to per-session buffer at `~/.gstack/insights/<repo-slug>/.buffer/$CLAUDE_SESSION_ID.jsonl` (session id from env). Session-boundary finalize triggers: (a) `claude --resume` boundary, (b) 30min idle, (c) explicit `/insight-flush`. **Finalize runs async in background** — Stop hook returns immediately; filter pipeline + upload happen out-of-band so user never observes capture latency. |
| **Routing latency budget** | UserPromptSubmit injection ≤500ms p95. **Hot path = local cache only (no git fetch in hot path)**. Retrieval = topic-tag exact match → fallback to embedding similarity. Embedding model = `sentence-transformers/all-MiniLM-L6-v2` bundled at plugin install via first-run `pip install` script (~50MB download, one-time). Empty-state behavior: if 0 lessons in cache → skip injection silently (no error to user). |
| **Topic-tag origin** | Topic tags inferred at capture time by **Layer 2 haiku call** (single pass, alongside privacy redaction). Topic is **routing metadata**, not solution authority — multiple lessons share a topic and are all retrieved as good/bad examples. P3 `no solution tag` rule applies to lesson body, not routing metadata. Topics drift over time → measured in Gate 1 via `/insight-rate` accuracy. |
| **Filter test corpus** | ≥500 real jsonl turns from founder's `~/.claude/projects/` history, manually labeled. Gate 0 pass = 0 PII leaks + ≤2% false positive. Founder labeling effort: 8-16 hours one-shot. |
| **Rating capture** | `/insight-rate <lesson-id> <good|bad|irrelevant>` slash command. Optional inline thumb prompt suppressed in v1 (preserves user-unaware default). Ratings written to mirror's `ratings.jsonl`. Gate 1 metrics computed from this file. |
| **Scaffolding disposition** | **KEEP**: 71 tests + 28 UAT + canned answers + statusline + force-install + marketplace. **EXTEND**: tests grow to cover capture/upload/filter/retrieval/rating pipeline (added as Step 4 deliverable). **REPURPOSE**: `/insight-add` `/insight-search` become silent variants triggered by hooks (legacy invocation still works for manual override). **NO DELETE**. |

### Open Questions (deferred to /plan-eng-review or v2)

1. Lesson decay/staleness policy — current draft: stamp lessons with `model_version + capture_date`; retrieval downweights `>6mo OR different model family`. Lock policy in /plan-eng-review.
2. Per-folder ACL (P4 v2) — required before any deployment to mixed-trust repos.
3. Topic inference robustness — drift risk if inference is wrong; embedding fallback covers this in v1 but should be measured in Gate 1.
4. Multi-org / multi-remote engineer — contractor in repo-A and repo-B with different teams. v1 install isolates per remote (P4); document in /plan-eng-review whether this is sufficient.
5. Auto-create of `<repo>-insights` mirror — does plugin auto-create on first push (requires admin scope), or must org admin pre-create? Current fallback: warn + manual `gh repo create`. Lock in /plan-eng-review.
6. **PAT rotation policy**: when an engineer leaves the team, founder revokes their PAT from GitHub UI; mirror history is immutable but write access is severed. v1 is manual; v2 may automate via GitHub App. Document in /plan-eng-review.
7. **Buffer crash recovery**: if Claude Code crashes mid-session, on next plugin startup scan `~/.gstack/insights/<repo-slug>/.buffer/` for orphan files older than 30min idle threshold and finalize them through the same pipeline (filter + upload). Lock idempotency guarantees in /plan-eng-review.
8. **Background clone timeout**: 5s budget on first UserPromptSubmit clone wait; on timeout, clone retries silently in background and retrieval is skipped for that prompt (user sees no error, no UI hint).

## Success Criteria

**Gate 0 (must pass before live capture writes any data)**: filter pipeline achieves **0 PII leaks + ≤2% false-positive** on 500-turn gold corpus from founder's own `~/.claude/projects/` history.

**Gate 1 (4-week B run after build complete)**:
- 4 V4team engineers run v2 plugin end-to-end for 4 weeks without manual intervention
- ≥1 cross-engineer lesson retrieval per week judged relevant (via `/insight-rate good`) by receiving engineer
- ≥1 documented loop-shortened incident in those 4 weeks (founder logs in `incidents.md` addendum)
- 0 PII leakage incidents in red-team audit (founder runs weekly red-team on uploaded jsonl sample, owner = founder, protocol = manual review of 50 random uploaded turns/week)
- Cost stays under $200/month for 4-engineer team (see Cost Model — $200 is ceiling with $120 reserve, target spend ≈$80)
- Routing latency p95 ≤500ms

**Kill condition (B's hard kill switch)**: abandon B at end of week 4 if **any** of:
- 0 cross-engineer relevant retrievals
- 0 loop-shortened incidents
- PII red-team finds any leak in audited sample
- Routing latency p95 > 1000ms

External-user adoption: **out of v1 scope**. Move to v2 success criteria post-Gate-1.

## Cost Model

Haiku 3.5: input $0.80/M tokens, output $4/M tokens.

**"Turn" definition**: 1 turn = 1 Stop hook firing = 1 complete agent response (potentially containing multiple tool calls within). Cost math below counts at this granularity.

Assumed load: 4 engineers × ~5 sessions/day × ~30 turns/session = ~600 turns/day team-wide. Average chunked turn = ~1k input tokens (chunking strategy mandatory — full 50-200k token sessions through haiku would 50-100x cost).

- Layer 2 haiku per-turn: ~1k in + ~200 out → ($0.80 × 0.001) + ($4 × 0.0002) = $0.0008 + $0.0008 = **$0.0016/turn**
- Daily team: 600 × $0.0016 ≈ **$1/day** ≈ $30/month for filter pipeline
- Retrieval-time agent calls (good/bad disambiguation when both examples returned): ~$50/month estimate
- Embedding (local sentence-transformers): $0
- Mirror repo storage (plain committed jsonl, no LFS): GitHub free tier covers; 4MB/day = 1.5GB/year per repo
- **Target spend ≈ $80/month**. **Ceiling = $200/month** with $120 reserve for retrieval-time bursts and unforeseen overruns.
