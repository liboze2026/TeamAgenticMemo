## Problem Statement

The prior design chose Approach B: build the full user-unaware, repo-scoped
precedent router. The architecture is now detailed enough. The missing artifact is
the four-week Gate 1 runbook that decides whether the bet is true in real teammate
use.

The hard truth: the founder skipped the prior assignment to observe one teammate for
30 minutes. That means Gate 1 must compensate by turning real incidents, ratings,
latency, and privacy audits into the source of truth.

## Demand Evidence

- Prior incident: one teammate lost roughly 8 hours in a repeated Claude Code loop
  that an existing founder lesson would likely have prevented.
- Current team: 4 V4team engineers, internal-only v1.
- Current product state: B-scope implementation exists in
  `/Users/m1/projects/insights-share/`.
- Current Gate 0 check, run on 2026-04-26:
  `bash tests/gate0/run-gate0.sh --strict` passed with 654 turns, 4 buckets,
  54 labeled spans, 0 leaks, 0 false positives, adversarial recall 1.0.

## Status Quo

Without Gate 1, the team can keep adding plugin code and tests while still not
knowing whether insights-share shortens real teammate loops. That is the classic
trap here: implementation progress without product truth.

## Target User and Narrowest Wedge

Target user: the 4 V4team engineers working inside trusted-org repos.

Narrowest wedge: during a real Claude Code prompt, the plugin injects a prior good
or bad example from another session, the receiving engineer rates it good, and the
team can point to a concrete loop that got shorter.

## External Baselines Used

- NIST AI RMF: official `.gov`; frames AI risk management as Govern, Map, Measure,
  Manage, with use-case profiles.
- NIST Privacy Framework: official `.gov`; frames privacy as ongoing risk
  management, not a one-time filter.
- OWASP Top 10 for LLM Applications: OWASP official project, GitHub repo around
  1.2k stars; relevant risks include prompt injection, sensitive information
  disclosure, excessive agency, and overreliance.
- Ragas metrics docs: project official docs, GitHub repo around 13.6k stars;
  useful split between retrieval quality, topic adherence, tool/agent goal
  accuracy, and feedback loops.

## Premises

1. Gate 1's main metric is cross-engineer loop shortened, not retrieval accuracy.
2. `/insight-rate good` is necessary but not sufficient. It proves local relevance,
   not product value.
3. Privacy audit is a kill gate. Any PII leak in uploaded jsonl ends the run.
4. UserPromptSubmit hot path p95 must remain <=500ms. Above 1000ms is a kill gate.
5. Because new teammate observation was skipped, weekly incident review is mandatory.
6. Four weeks is a hard stop. If the kill gates fire, reverse migration starts.

## Approaches Considered

### Approach A: Minimal Trial Log

Track weekly ratings, one loop-shortened incident, one privacy sample, and latency.
Effort: S. Risk: medium.

Pros:
- Fastest to run.
- Low teammate burden.

Cons:
- Too little evidence to diagnose failure.
- Latency and incident definitions can drift.

### Approach B: Operating Runbook

Define owner, cadence, metrics, privacy audit, incident review, kill conditions, and
reverse migration trigger for the four-week run. Effort: M. Risk: low-medium.

Pros:
- Hard enough to kill the bet.
- Light enough for a 4-person internal trial.
- Reuses existing code: `ratings.jsonl`, mirror jsonl, Gate 0, `tests/run.sh`,
  `/insight-rate`, and hot-path latency checks.

Cons:
- Founder must run weekly incident review and privacy sampling.
- The skipped teammate observation remains visible as a risk, not hidden.

### Approach C: Full Experiment Protocol

Add baseline week, incident taxonomy, cohort rubrics, weekly report templates, and a
final statistical memo. Effort: L. Risk: high friction.

Pros:
- Best truth quality.
- Creates an external-ready evidence package.

Cons:
- Too heavy for the current 4-person internal run.
- Fights the user-unaware thesis by turning usage into paperwork.

## Recommended Approach

Use Approach B. A is too thin, C is too much process. B tests the founder bet with
four evidence streams: relevance, loop shortening, privacy, and latency.

## Gate 1 Runbook

### Owner

Founder owns Gate 1. Teammates only need to use the plugin normally and run
`/insight-rate <lesson-id> good|bad|irrelevant [reason]` when an injected lesson is
clearly useful or not useful.

### Duration

Run for 4 calendar weeks after all 4 engineers have installed the plugin in the same
trusted-org repo and the mirror repo is syncing.

### Daily Checks

- Confirm each active repo has local insight cache at `~/.gstack/insights/<slug>/`.
- Check `ratings.jsonl` for new ratings.
- Check that no teammate reports visible slowdown during prompt submit.
- Do not ask teammates to tag lessons manually. That would break the user-unaware
  thesis.

### Weekly Checks

1. Export ratings summary:
   - count of `good`
   - count of `bad`
   - count of `irrelevant`
   - distinct receiving engineers
   - distinct lesson IDs
2. Pick 50 random uploaded turns from mirror jsonl and manually review for PII.
3. Record at least one incident review:
   - what the teammate was trying to do
   - which lesson was injected
   - what action changed because of it
   - time saved estimate
   - whether the lesson came from another engineer
4. Run a hot-path latency sample using the existing B6 pattern in `tests/run.sh`.
5. Write a one-page weekly memo under the mirror repo or `~/.gstack/projects/`.

### Success Criteria

Gate 1 passes only if all are true:

- 4 engineers run the plugin for 4 weeks without manual intervention.
- At least 1 cross-engineer retrieval per week is rated `good`.
- At least 1 documented loop-shortened incident occurs during the 4 weeks.
- Weekly 50-turn PII audits find 0 leaks.
- UserPromptSubmit hot path p95 remains <=500ms.
- Total estimated operating cost stays under $200/month for the 4-engineer team.

### Kill Conditions

Kill the B bet at week 4 if any are true:

- 0 cross-engineer relevant retrievals.
- 0 loop-shortened incidents.
- Any PII leak in audited uploaded jsonl.
- UserPromptSubmit p95 exceeds 1000ms.
- Any teammate disables the plugin because it disrupts normal Claude Code work.

Do not extend the run. Extension is how bets turn into sunk cost.

### Reverse Migration
