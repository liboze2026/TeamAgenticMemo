## 一、单句目标

> **让 TeamAgent 自动从开发过程中的错误信号里提炼规则候选，经人工逐条确认后写入知识库，形成闭环飞轮。**

---

## 二、背景与调研摘要

### 行业现状（2026-04）

- 目前没有任何 coding agent 能真正自动从错误生成规则——Cursor/Devin 等均需人工写 `.cursorrules` / `agents.md`
- 2026 年竞争焦点：谁能让"检测失败 → 生成规则"飞轮自动化
- 最值得借鉴的学术设计：
  - **ETGPO**（arxiv:2602.00997）：失败 trace 批量分类 → 频率过滤 → 每类生成规则 → 追加系统提示
  - **EvolveR**（arxiv:2510.16079）：成功轨迹提"指导原则"，失败轨迹提"警告原则"，动态评分淘汰低质规则
  - **Letta 睡眠期 Agent**：任务完成后异步整理，不阻塞主流程

### 现有基础设施

- `analyze --commit`：已能检测纠正时刻 → LLM 提取 → 写 KnowledgeStore → 校准
- `PersistedEvent`：已落盘 `hook-post.result`、`ai.override.ignored/complied`、`hook-pre.blocked`
- `KnowledgeEntry`：完整 tier/confidence/demerit/category(C/E/S/K) 体系
- `ruleBasedCorrectionDetector`、`llmBasedKnowledgeExtractor`：可直接复用

### 当前缺口

- `analyze` 是手动的、单 session 的，无跨 session 聚类
- 无错误信号自动采集（build 失败、AI override 绕过等）
- 无候选规则暂存与交互审核工作流

---

## 三、设计哲学

1. **最大化复用**：`llmBasedKnowledgeExtractor`、`DualLayerStore`、`runCalibrationPipeline` 等直接复用，不造轮子
2. **Functional Core, Imperative Shell**：`signal-filter`、`cross-session-cluster`、`error-batch-builder` 全是纯函数，IO 在 shell 层
3. **两种模式**：`efficient`（四层 token 节省）和 `full`（全量 LLM 扫描）并存，`--mode` 切换
4. **人工审核门**：规则候选暂存队列，`review-candidates` 交互式逐条确认，不自动写入
5. **Port 先行**：新增 Port 必须先写契约测试再写实现

---

## 四、错误信号范围

| 信号 ID | 来源 | 描述 | 采集方式 |
|---------|------|------|---------|
| A | session 日志 | 用户纠正 AI（correction_moment） | `ruleBasedCorrectionDetector`（已有）|
| B | events.db | build/test 失败（hook-post.result, succeeded=false） | `SqliteEventLog` 读取 |
| C | events.db | AI override 被人类强行覆盖（ai.override.ignored） | `SqliteEventLog` 读取 |
| D | session 日志 | 同一任务多次连续失败（multi_failure） | session 解析 |
| G | events.db | hook-pre.blocked 后用户继续执行（规则被绕过） | `SqliteEventLog` 读取 |
| H | 跨 session | 同一 tag/pattern 在 ≥N 个不同 session 重复出现 | `crossSessionCluster` 聚类 |

## 五、整体架构

```
Trigger（cron B / CI hook C / Stop hook）
        ↓
scan-errors [--mode efficient|full] [--since <duration|ISO>]
        ↓
ErrorSignalCollector.collect(since)
  ├─ A: ruleBasedCorrectionDetector.detect(sessions[])
  ├─ B: eventLog → hook-post.result{succeeded:false}
  ├─ C: eventLog → ai.override.ignored
  ├─ D: sessions[] → multi_failure
  ├─ G: eventLog → hook-pre.blocked（无后续纠正）
  └─ H: crossSessionCluster(allSignals)
        ↓ RawErrorSignal[]

── efficient mode 专用 ──────────────────────
SignalFilter.filter(signals[])
  ├─ 去重（same knowledge_id + signal type → 保留最新）
  ├─ 频率门控（跨 session 出现 < MIN_FREQ → 丢弃）
  └─ 权重阈值（weight < 0.3 → 丢弃）
─────────────────────────────────────────────
        ↓

ErrorBatchBuilder.build(signals[])
  └─ 按 category(C/E/S/K) 分组 → BatchPrompt[]

        ↓ 单次/少量 LLM 调用

llmBasedKnowledgeExtractor（复用，换 prompt）
  → KnowledgeEntry[] candidates

        ↓
CandidateQueue.enqueue(candidates[])   ← SQLite 暂存
bus.emit({ kind: "error.candidate.added" })

        ↓（用户手动触发）
review-candidates（交互式 a/r/e/s/q）
        ↓ approve
store.add → runCalibrationPipeline → recompile CLAUDE.md
bus.emit({ kind: "error.candidate.approved" })
```

### 6.1 新增 Port（先写契约测试）

**`packages/ports/src/error-signal-collector.ts`**

```typescript
export interface RawErrorSignal {
  id: string;
  signalType: "A" | "B" | "C" | "D" | "G" | "H";
  weight: number;
  sessionIds: string[];          // 来源 session（H 信号可能多个）
  context: string;               // 供 LLM 提取用的原文上下文
  suggestedCategory?: "C" | "E" | "S" | "K";
  timestamp: string;
}

export interface ErrorSignalCollector {
  collect(since: Date): Promise<RawErrorSignal[]>;
}
```

**`packages/ports/src/candidate-queue.ts`**

```typescript
export interface RuleCandidate {
  id: string;
  entry: KnowledgeEntry;          // 候选 KnowledgeEntry（未写入 store）
  sourceSignals: string;          // 来源信号摘要（展示用）
  status: "pending" | "approved" | "rejected" | "skipped";
  created_at: string;
  reviewed_at?: string;
}

export interface CandidateQueue {
  enqueue(candidates: Omit<RuleCandidate, "status" | "created_at">[]): void;
  listPending(): RuleCandidate[];
  updateStatus(id: string, status: RuleCandidate["status"]): void;
  count(): number;
}
```

## 七、数据存储

### CandidateQueue SQLite schema

```sql
CREATE TABLE IF NOT EXISTS rule_candidates (
  id          TEXT PRIMARY KEY,
  entry_json  TEXT NOT NULL,       -- JSON 序列化的 KnowledgeEntry
  source_signals TEXT NOT NULL,    -- 来源信号摘要（展示用）
  status      TEXT NOT NULL DEFAULT 'pending',
  created_at  TEXT NOT NULL,
  reviewed_at TEXT
);
```

存储路径：`~/.teamagent/candidates.db`（与 `events.db` 同目录，独立文件）

## 八、CLI 详细规格

### `teamagent scan-errors`

```
teamagent scan-errors [options]

Options:
  --mode <efficient|full>   默认 efficient
  --since <duration|ISO>    扫描窗口（如 24h, 2026-04-15T00:00:00Z）
                            默认读 ~/.teamagent/scan-state.json 上次时间戳
  --min-freq <N>            efficient mode 频率门控（默认 2，CI 建议 1）
  --quiet                   无候选时静默退出；有候选时打印提示
  --dry-run                 只打印候选，不写入队列
```

### `teamagent review-candidates`

```
teamagent review-candidates [options]

Options:
  --limit <N>   每次展示条数（默认全部）

交互操作：
  [a] approve — 写入 KnowledgeStore，触发 calibrate + recompile
  [r] reject  — 标记 rejected，不写入
  [e] edit    — 打开 $EDITOR 编辑候选内容后再 approve
  [s] skip    — 留在队列，下次再看
  [q] quit    — 退出，未处理条目保留
```

## 九、触发机制

### B：Stop hook（每次 Claude Code 会话结束后）

在 `settings.json` 里配置：

```json
{
  "hooks": {
    "Stop": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "pnpm teamagent scan-errors --mode=efficient --since=2h --quiet"
      }]
    }]
  }
}
```

### C：CI 集成

```yaml
# .github/workflows/ci.yml
- name: TeamAgent error scan
  if: failure()
  run: pnpm teamagent scan-errors --mode=full --since=1h --min-freq=1
```

### 扫描状态持久化

`~/.teamagent/scan-state.json`：

```json
{
  "lastScanAt": "2026-04-16T02:00:00Z",
  "lastScanMode": "efficient"
}
```

## 十、Token 节省策略（efficient mode）

| 层 | 机制 | 预期效果 |
|----|------|---------|
| 1. 规则粗筛 | 去重 + 权重阈值(0.3) | 丢弃约 50% 低权重信号 |
| 2. 频率门控 | 跨 session ≥ MIN_FREQ | 丢弃约 30% 偶发信号 |
| 3. 批量合并 | 按 C/E/S/K 分组 | 最多 4 次 LLM → 通常 1-2 次 |
| 4. 增量窗口 | `--since` 只处理新信号 | 避免重复处理历史 |

## 十一、错误分类

- **大类**：复用现有 `KnowledgeEntry.category`（C/E/S/K）
- **细分**：`tags` 字段由 LLM 自由生成（不预设枚举）
- **聚类（H 信号）**：`crossSessionCluster` 对 `tags` 做字符串归一化后频率统计
