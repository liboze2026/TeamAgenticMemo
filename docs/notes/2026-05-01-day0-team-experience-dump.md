## Ⅰ. 老队友踩过的 40 个坑

### A · Git / Commit / Branch / PR (1–10)

1. `git push --force` 不看分支 — 别人 commit 直接被覆盖消失 → 永远 `--force-with-lease`。
2. commit message 写 "fix bug" / "update" — 事后完全不知道改了啥 → imperative：`feat/add user validation`。
3. PR 不写 description 直接丢链接 — reviewer 没上下文懒得看 → 必须有：做了什么 / 为什么 / 怎么验证。
4. 大 commit 塞所有改动 — review 无法细粒度审、revert 一丢全丢 → 按逻辑拆原子 commit。
5. 长期开 feature 分支不同步 main — 合并冲突爆炸 → 每天或隔天 `git rebase main`。
6. 直接在 main 分支 hotfix — 没测试环境验证就上线 → hotfix 也走 PR + review，cherry-pick 到 prod。
7. merge 不加 `--no-ff` — 历史变成蜿蜒 merge 节点 → 保结构用 `--no-ff`，干净史用 rebase。
8. `.gitignore` 改完不生效 — 文件已 staged → `git rm -r --cached .` 再加。
9. 多人协作不用 protected branch — main 可以被强推 → 设 PR + review + status check。
10. stash 不给名字 — 堆满 `stash@{0}` 不知谁是谁 → `git stash push -m "name"`。

### B · Code Review / Testing / TDD (11–20)

11. 覆盖率 80% 但只量不质 — 行数刷出来核心分支裸奔 → 盯分支/路径覆盖。
12. TDD 红绿反转乱序 — 测试成了写完代码后的文档 → 严格 红 → 绿 → 重构。
13. Code review 只挑风格 — 业务 bug 没人管 → 风格交给 linter，review 盯正确性/安全/可维护。
14. 覆盖率不看 diff — 整体好看，新代码全 0% 也 merge → 每次 PR 看增量覆盖率。
15. Mock 套娃 — 真实调用链一改全挂 → Mock 最小化，核心逻辑用真协作对象做集成测试。
16. 测试数据硬编码 — 改业务字段全红 → 用 factory / fixture。
17. 没人 check 测试质量 — 假测试混进去 → reviewer 有义务质疑测试逻辑。
18. 滥用 `/* istanbul ignore */` 凑覆盖率 → CI 禁止滥用，必须配 ticket。
19. E2E 当单元跑 — 改小功能等半小时 → 金字塔分层，E2E 只保关键路径。
20. "以后再补测试"成习惯 → DoD 必须含测试，无测试不算完成。

### C · 上线 / 部署 / 灰度 / 监控 / 回滚 (21–30)

21. 灰度无梯度 — 流量突增 bug 集中爆 → 5%→15%→50%→100% + 每级人工确认。
22. 回滚脚本没进 CI — 灾难时刻敲错命令 → 回滚脚本必须像部署脚本一样进 CI。
23. 监控只配 P99 — 服务烂一半 P99 还"健康" → P50/P90/P99 三档都要，SLO 按 P50 报。
24. 健康检查依赖下游 — k8s 认健康，流量进来 OOM → 健康检查只测进程存活。
25. 灰度变量与 Feature Flag 混用 — 实验数据被开关污染 → 二者隔离。
26. 回滚没检查 Schema 兼容 — 老版本读不到新字段 → Schema 变更必须向后兼容。
27. 监控大盘塞太多图 — oncall 半夜翻三屏 → oncall 大盘只放 QPS/Error/Latency 三张图。
28. 灰度期关链路追踪 — 出问题靠猜 → 采样率可降但不能关，建议 ≥10%。
29. 发布选高峰期 — 出错时用户当场感知 → 发布永远低峰期。
30. 没数据回填预案 — 业务回滚但新字段数据残留导致 crash → Schema 变更同步打回填脚本。

### D · 协作 / 沟通 / 估时 / 文档 / On-call / Onboarding (31–40)

31. 估时只报 best case → 三点估时 × 1.3。
32. 文档写给"未来的自己" → 设计文档强制含 "Why we chose X over Y"。
33. On-call 预案只写正向流程 → runbook 末尾强制附"历史事故栏"。
34. 新人 onboarding 丢链接自学 → 第一天当面过任务，第二周起每天 15 分钟 pair。
35. 估时不留 buffer → 对外 deadline 加两周 buffer，内部用压缩工期排期。
36. 文档评审只说"这里有错" → 评审意见带 P0/P1 优先级 + 修改方向。
37. 任务"谁闲谁接"不看技能图谱 → 用技能矩阵分配关键路径。
38. 会议有议程无结论 → 每个 agenda 必须有结论 + owner + due date。
39. 新人 PR 通过即合并不 walkthrough → 合并前 15 分钟讲"为什么这样写"。
40. On-call 交接只口头说一句 → 交接文档结构化：阈值 / 最近根因 / 应急联系人。

## Ⅱ. 10 条最重要的团队判断标准

1. **上下文所有权** — PR/评审先确认决定权，缺席指代理人；无人认领默认阻塞。
2. **承诺颗粒度** — 估时以半天为最小单位，无法估时先说"不确定"。
3. **阻塞可视化** — 阻塞 >1 天必须升级，私下等待视为蔓延责任。
4. **契约优于默契** — 跨团队接口必须书面确认，口头协议等于没有。
5. **增量即交付** — 任何任务以可演示增量为准，写完代码 ≠ 完成。
6. **悲观估时法** — 估时 × 1.5，乐观估算不进计划。
7. **缺席即默认同意** — 评审迟到 ≥15 分钟视同弃权，不得事后翻案。
8. **失败即学习** — 任何事故 / missed estimate 必须 post-mortem。
9. **依赖先验性** — 跨团队依赖任务开始前确认，开发中发现 = 风险未管控。
10. **退出条件先行** — 任务开始前写明 DoD，未对齐不动工。

## Ⅲ. 5 个典型失败案例

| # | 背景 | 错误决策 | 爆发与损失 | 根本教训 |
|---|------|----------|-----------|----------|
| 1 | 核心模块作者单飞，无文档 | 「代码即文档」 | 半年后维修耗时 6×，2 次 P0 | 知识不编码 = 零，人员流动是熵增加速器 |
| 2 | OKR 季末冲刺，测试时间被挤 | 「牺牲测试换进度」 | 上线后核心链路故障，订单损失 800 万+ | 质量不是 phase，欠债必还且利息极高 |
| 3 | 技术债积压 2 年，重构提议被拒 | 「能用就不动」 | 小变更触发循环依赖，雪崩故障 3 天 | 技术债是复利，越晚还代价越大 |
| 4 | 招聘季为 HC 录用面试强者 | 「进来再培养」 | 6 个月后协作摩擦致 2 名资深离职 | 招聘是逆向选择，看走眼代价成倍 |
| 5 | 多数人熟悉微服务，新系统选型 | 「用熟悉技术解决新问题」 | 分布式事务/网络/运维远超预期，延期 4 月 | 先问问题域特征，再匹配技术 |

## Ⅳ. 总结：Day 0 倒经验的最佳实践

1. **写作要点**：坑 = 场景 + 决策 + 后果 1 行结论；标准 = 触发 + 动作 + 验收三段；
   失败案例 = 背景 + 错误 + 爆发 + 教训四行定格。写完自问"后人能直接操作吗"。
2. **归类与索引**：三级结构 = 维度标签（角色/阶段/风险）+ 动词化标题（避免/必须/不要）
   + 时间戳前缀。按使用频率排序，最近在前，久未触达降级或归档。
3. **维护节奏与失效信号**：「已不适用未标注 / 多次匹配未执行 / 跨版本无变更记录」
   任意一条出现立刻重写或归档。季度做减法（合并重复、删除过时），不让知识库变沼泽。
