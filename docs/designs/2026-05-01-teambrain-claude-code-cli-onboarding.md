## Problem Statement

TeamBrain's first internal product value is not generic team knowledge management. It is replacing the repeated 45-minute team experience-sync meeting with a Claude Code CLI runtime experience that installs correctly, injects the right team lessons at the right time, and proves what happened with evidence.

The team already has real usage and internal sponsor interest, but adoption is blocked by three practical failures:

- Teammates do not know how to install it.
- Teammates install it into the wrong surface, such as Codex CLI or Claude Code App, while the product is designed for Claude Code CLI.
- Teammates do not fully trust what it installed, what it injected, or how to debug/undo it.

## Demand Evidence

- Current experience sharing costs about 45 minutes across 5 people, roughly 225 minutes of team time per sync.
- Slack group messages are used, but few people read them, so Slack is not a reliable knowledge delivery channel.
- New teammate onboarding currently adds about 2 hours of avoidable time.
- The sponsor pain is not "AI memory is cool"; the pain is wasted team time and repeated explanation.

## Status Quo

```
Team lead / teammate knows a lesson
  |

  |

  |

  +--> does not trust whether it worked
  |

```

This workflow is expensive because the same expert knowledge has to be re-spoken instead of becoming a runtime constraint inside Claude Code CLI.

## Target User & Narrowest Wedge

Primary user: a new teammate or existing teammate using Claude Code CLI inside the TeamBrain repo.

Buyer/sponsor: the team lead who wants to stop spending team meeting time re-explaining setup, conventions, and known failure modes.

Narrowest wedge:

```
team onboarding
  |
  v
detect Claude Code CLI surface
  |
  +--> wrong surface: block, explain, show correct path
  |
  v
install TeamBrain hooks and team experience pack
  |
  v
open a Claude Code CLI session
  |
  v
prove injected experience with judge JSON
```

## Constraints

- Focus only on Claude Code CLI usage experience for the first wedge.
- Do not claim support for Codex CLI or Claude Code App in the initial product path.
- Ordinary users should not have to understand background sync failures, but logs and judge evidence must show `fail`, `needs_review`, or `evidence_missing` clearly.
- The system cannot depend on one champion constantly explaining how to fix it.
- All feature verification must be both agent-verifiable and human-by-hand verifiable.

## Premises

1. TeamBrain's first internal value is replacing repeated 45-minute experience-sync meetings with Claude Code CLI runtime guidance.
2. The first demo must start at onboarding: detect Claude Code CLI, reject wrong surfaces, install hooks, and show proof.
3. Trust requires visible evidence: what was installed, what was injected, what matched, what failed, how to undo.
4. If no natural owner exists, the system must become self-service: onboarding should install it, and failure evidence should be readable by Claude Code.
5. The long-term team-memory plan is still right, but it should be sequenced after the onboarding trust layer.

## Landscape Notes

Agent-memory products already make "shared memory for coding agents" a visible category, but TeamBrain's useful wedge is narrower: meeting experience becomes Claude Code CLI runtime behavior with proof. Local-first workflow and onboarding research point the same way: expert knowledge must live inside the developer workflow, with observability and reusable human-curated artifacts, not only in a wiki.

## Approaches Considered

### Approach A: Meeting-to-Rules Minimum Loop

Convert one 45-minute meeting transcript or recording into a Claude Code CLI readable team experience pack. Demonstrate that the next teammate session receives relevant injected guidance and avoids one repeated mistake.

Effort: M. Risk: Low-Med.
Pros: hits sponsor meeting-time pain; creates a clear before/after demo; reuses hooks, compiler, and judge-harness patterns.
Cons: does not fully solve install trust; transcript-to-rule quality still needs review.

### Approach B: Onboarding-First Trust Layer

Build a focused onboarding flow that detects whether the user is actually in Claude Code CLI, blocks wrong surfaces, installs the hook and team experience pack, and produces evidence plus rollback/debug instructions.

Effort: M. Risk: Low.
Pros: fixes observed A/B/D onboarding failures; makes the system usable without the original champion; gives the sponsor a concrete adoption path.
Cons: does not by itself prove meeting experience extraction; can look like infrastructure unless paired with a meeting-to-rules demo.

### Approach C: 14-Day Ideal Architecture

Execute the full team-memory plan: audio/transcript ingestion, PII redaction, problem clusters, multiple variants, git sync, SessionStart import, runtime injection, judge evidence, and claim report.

Effort: L/XL. Risk: Med-High.
Pros: matches long-term TeamBrain shape; covers meeting reduction, onboarding, repeat-error prevention, and evidence; creates future team-memory infrastructure.
Cons: large for a 14-day plan; if onboarding/trust fails, users may never reach the deeper value.

## Recommended Approach

Do all three, but sequence them instead of treating them as parallel scope:

1. **B first: Onboarding Trust Layer**
   Make the first user minute correct. Detect Claude Code CLI, reject wrong surfaces, install hooks, show evidence, and expose rollback/debug.

2. **A second: Meeting-to-Rules Demo**
   Use one real or sanitized meeting/transcript sample to create a team experience pack and show it being injected into a teammate session.

3. **C third: Expand into 14-Day Team Memory**
   After the first two demos work, continue into the full architecture: PII, cluster/variant governance, sync, claim data, and third-party judge harness.

This preserves the ambition of the 14-day plan while reducing the biggest adoption risk: users failing before they ever experience the memory layer.

## Success Criteria

- A new teammate can run one onboarding path and end in a correctly configured Claude Code CLI environment.
- Wrong surfaces are detected and rejected with a clear explanation.
- A human can verify installation by hand, and an agent can verify the same facts from JSON evidence.
- One meeting-derived experience appears in a later Claude Code CLI session.
- Judge evidence records source, install status, injection status, match status, and failure states.
- Team lead can see at least one metric tied to the original pain: meeting minutes avoided, onboarding time reduced, repeat-error prevented, or evidence completeness.

## Distribution Plan

The deliverable is a CLI/onboarding path inside TeamBrain, not a separate web service.

Distribution path:

- Package through the existing `teamagent` CLI/npm bundle path.
- Team onboarding points new teammates to one command or one checked-in onboarding document.
- Install command configures Claude Code CLI hooks only after environment detection passes.
- `teamagent doctor --json` becomes the support contract for both humans and Claude Code.
- `teamagent bug-report --bundle` gives the teammate's Claude Code enough evidence and source context to debug failure.

CI/CD:

- Keep the existing package/test pipeline.
- Add fixed judge harness runs for onboarding, wrong-surface rejection, meeting-to-rules injection, and evidence completeness.
- Do not mark the demo complete unless both human-by-hand verification and agent-readable JSON verification pass.

## Dependencies

- Existing hook install path and doctor command.
- Existing or planned bug-report bundle.
- Existing CLAUDE.md compile / runtime injection path.
- PII redaction before any meeting transcript is synced or persisted.
- A small sanitized meeting/transcript sample for the first demo.

## The Assignment

Before implementation, run one live observation with a teammate:

1. Ask them to start from the current team onboarding material.
2. Watch silently until they either install correctly, install into the wrong surface, or lose trust.
3. Record the exact failed step and the exact words they use when confused.
4. Use that observation to choose the first `teamagent onboarding` acceptance test.

The goal is not more planning. The goal is one observed install failure that becomes the first fixed demo path.
