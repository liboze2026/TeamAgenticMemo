# Team Memory 重设计：非冲突多解并列治理（2026-04-30）

## 背景

旧设计把“规则冲突裁决”作为核心流程，默认假设同一问题需要收敛到单一答案。
在当前产品方向下，这个前提不成立：

- 同一个问题可能存在多种都可行的解法；
- 用户给出的经验应作为 **good example**（优秀示例），而不是唯一标准答案；
- 系统应优先做“场景化并列展示 + 适用条件解释”，而不是“二选一下线”。

因此本次设计从“冲突治理”切换到“多解治理”。

---

## 设计目标

1. **不再使用冲突作为主范式**
   - 不把不同解法标为互斥。
   - 不要求强制保留一条、下线一条。

2. **同题多解并列**
   - 对同一问题簇（problem cluster）可同时保留多条解法。
   - 编译/注入时并列输出，按“适用条件”排序，而非“唯一正确”。

3. **用户经验定位为 good example**
   - 用户提供的规则默认为高价值示例（example）。
   - 仅在“明确错误/有害”时进入下线流程；不是因为“和别的答案不同”就下线。

4. **治理动作从裁决改为注释与分层**
   - 主要动作：补充适用上下文、增加 trade-off、分层推荐。
   - 次要动作：合并重复项、归档过时项。

---

## 新术语

- **Problem Cluster**：语义上同一类问题。
- **Solution Variant**：该问题下的一种可行解法。
- **Good Example**：用户提供且被验证“可工作”的示例，作为 variant 的证据源。
- **Applicability**：适用条件（技术栈、规模、性能约束、团队偏好、风险等级）。

---

## 数据模型调整（概念层）

原 `conflict_with` 语义降级为历史兼容字段，不再驱动治理。
新增（或逻辑上引入）以下概念字段：

- `problem_cluster_id`: string
- `variant_id`: string
- `applicability`: object[]（可多条）
- `tradeoffs`: string[]
- `evidence_kind`: `user_example | benchmark | incident | doc`
- `is_user_good_example`: boolean

说明：
- 同一 `problem_cluster_id` 下允许多个 active variant。
- 不同 variant 可同时进入推荐输出。

---

## 工作流重构

### 1) 录入（ingest）

- 新经验进入后先做语义聚类：归入某个 `problem_cluster_id`。
- 若已存在同类解法：
  - 默认新增为并列 variant；
  - 自动补齐“与现有方案差异点”（不是冲突判断）。

### 2) 校验（validate）

- 保留客观校验：字段合法、引用路径合法、示例可复现。
- 去掉“不同解法必冲突”的判断逻辑。

### 3) 编译（compile）

- 对同一 cluster 输出 1~N 个 variant。
- 排序依据：
  1. applicability 命中度
  2. 近期成功率
  3. 证据新鲜度
- 输出格式中明确：
  - “还有其他可行解法”
  - 每条的适用条件与 trade-off

### 4) 运行时注入（inject）

- 当命中某 cluster：展示“并列建议卡片”，而非“唯一动作指令”。
- 高风险场景可保留 block 规则，但 block 针对“明确错误行为”，不针对“另一种正确解法”。

### 5) 生命周期（governance）

- **保留**：可工作、可解释、仍有适用场景。
- **合并**：表达重复、差异不足（合并证据，不删除多样性）。
- **下线**：仅在明确错误/失效/有害时触发。

---

## 用户经验策略

默认策略：

1. 用户新增经验 → `is_user_good_example = true`
2. 初始不收敛为唯一答案
3. 至少保留一个“用户样例视角”的 variant
4. 若系统已有不同方案：并列展示，并补“何时选A/何时选B”
