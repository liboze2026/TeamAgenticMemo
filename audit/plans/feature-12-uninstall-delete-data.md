# Feature #12 `teamagent uninstall --delete-data` 非自证 audit 草案

## 目标

验证 `teamagent uninstall --delete-data` 真实从 CLI 入口执行时，能在临时 `cwd` / `HOME` 中完成以下行为：

- 删除项目级数据目录 `./.teamagent`。
- 删除用户级数据目录 `~/.teamagent`，其中 `~` 必须来自测试注入的临时 `HOME`，不是审计者真实 home。
- 从 `CLAUDE.md` 删除 `<!-- TEAMAGENT:START ... -->` 到 `<!-- TEAMAGENT:END -->` 的 TeamAgent 自动区块。
- 保留 `CLAUDE.md` 中用户手写内容。
- 从 `.claude/settings.local.json` 删除 TeamAgent tagged hooks。
- 保留 `.claude/settings.local.json` 中用户 hooks 和其他用户配置。
- 使用外部 `find` / `diff` / Node JSON parser 验证结果，不以 TeamAgent 自身测试或内部 helper 的返回值作为证据。

## 源码追踪

入口链路：

- `package.json` 的脚本 `teamagent` 指向 `tsx packages/cli/src/bin.ts`。
- `packages/cli/src/bin.ts` 在 `case "uninstall"` 中调用：
  - `parseUninstallArgs(rest)`：只识别 `--delete-data` 和 `--dry-run`。
  - `uninstall(opts)`：执行真实卸载逻辑。
  - `renderUninstallResult(r)`：只负责终端输出，不参与文件修改。
- `packages/cli/src/commands/uninstall.ts`：
  - `cwd = opts.cwd ?? process.cwd()`，CLI 真实执行时依赖当前工作目录。
  - `home = opts.homeDir ?? os.homedir()`，CLI 真实执行时依赖 `HOME`。
  - 若 `.claude/settings.local.json` 存在，则调用 `uninstallHook({ cwd })`。
  - 若 `CLAUDE.md` 存在，则调用 `stripTeamagentBlock()` 删除 TeamAgent block；若删除后只剩空白，会直接删除 `CLAUDE.md`。
  - 若 `opts.deleteData` 为真，则删除 `path.join(home, ".teamagent")` 和 `path.join(cwd, ".teamagent")`。
  - 若未传 `--delete-data`，只追加“保留知识数据”的 action，不删除数据目录。
- `packages/cli/src/commands/install-hook.ts` 的 `uninstallHook()`：
  - 从 `.claude/settings.local.json` 删除以下 `_teamagentTag`：
    - `teamagent-pre-tool-use`
    - `teamagent-post-tool-use`
    - `teamagent-user-prompt-submit`
    - `teamagent-stop`
  - 若 `statusLine._teamagentTag === "teamagent-statusline"`，删除 `statusLine`。
  - 对没有 TeamAgent tag 的用户 hooks 保留。
  - 对其他顶层配置保留。

现有自证覆盖：

- `packages/cli/src/__tests__/uninstall.test.ts` 直接调用 `uninstall({ cwd, homeDir, deleteData })`，覆盖数据删除、CLAUDE block 删除、dry-run 等。
- `packages/cli/src/__tests__/install-hook.test.ts` 直接调用 `uninstallHook({ cwd })`，覆盖 tagged hook 删除和用户 hook 保留。
- 本 audit 需要补上“真实 CLI 入口 + 外部造数据 + 外部验结果”的非自证证据。

## 关键 JSON / 文件形状

审计用例必须由 shell 外部创建以下文件，不使用 `teamagent init` 或 `installHook()` 生成：

- `$PROJ/.teamagent/...`：项目级 TeamAgent 数据，运行后应不存在。
- `$HOME_DIR/.teamagent/...`：用户级 TeamAgent 数据，运行后应不存在。
- `$PROJ/CLAUDE.md`：包含用户手写内容和 TeamAgent block，运行后只剩用户内容。
- `$PROJ/.claude/settings.local.json`：
  - `PreToolUse` / `PostToolUse` / `UserPromptSubmit` / `Stop` 中同时放 TeamAgent tagged entry 和用户 entry。
  - `Notification` 放一个用户 hook，证明未知 hook 类型不会被误删。
  - `statusLine` 放 TeamAgent tagged statusLine，证明 tagged statusLine 会被删除。
  - `permissions` / `env` 放用户配置，证明非 hooks 配置保留。

## 非自证验证命令

从仓库任意目录执行。命令会创建临时目录，不修改当前项目数据；真实 CLI 的 `process.cwd()` 是 `$PROJ`，真实 `os.homedir()` 来自 `$HOME_DIR`。

test -x "$TSX" || {

  exit 2

ROOT="$(mktemp -d -t teamagent-f12-uninstall.XXXXXX)"

## 预期输出

关键输出应包含：

```text
== before data files ==
.../home/.teamagent/global/knowledge.jsonl
.../home/.teamagent/personal/knowledge.jsonl
.../proj/.teamagent/team/nested/knowledge.jsonl
== run real CLI ==
TeamAgent Uninstall
  已移除 hook 注册: .../proj/.claude/settings.local.json
  已从 CLAUDE.md 移除 TEAMAGENT 区块
  已删除: .../home/.teamagent
  已删除: .../proj/.teamagent
== external semantic JSON assertions ==
JSON_ASSERTIONS_PASS
PASS feature-12 uninstall --delete-data non-self audit
```

`diff -u` 不应输出差异；`find "$HOME_DIR" "$PROJ" -name .teamagent` 的结果文件应为空。

## 判定标准

通过：

- 命令整体 exit code 为 `0`。
- `data.before.find` 至少包含 3 个数据文件，证明测试确实先创建了用户级和项目级数据。
- `data.after.find` 为空，证明 `$HOME_DIR/.teamagent` 和 `$PROJ/.teamagent` 都不存在。
- `diff -u "$ROOT/CLAUDE.expected.md" "$PROJ/CLAUDE.md"` 无差异。
- `diff -u "$ROOT/settings.expected.json" "$SETTINGS"` 无差异。
- Node JSON parser 输出 `JSON_ASSERTIONS_PASS`。
- CLI 输出包含 hook 清理、CLAUDE block 清理、两个数据目录删除的 action。

失败：

- 任一 TeamAgent `_teamagentTag` 仍残留。
- 任一用户 hook 被删除或命令被改写。
- `permissions` / `env` 等用户配置丢失。
- `CLAUDE.md` 中 TeamAgent block 残留，或用户手写内容缺失。
- 任一 `.teamagent` 数据目录仍存在。
- CLI 不是从 `$PROJ` cwd 执行，或没有注入临时 `HOME`。

## 额外边界建议

本 feature 的主证据以上述端到端命令为准。若要补边界，可以再加一个极小用例：`.claude/settings.local.json` 只有 TeamAgent tagged `statusLine`、没有 `hooks` 对象时运行 `uninstall`。源码中 `uninstallHook()` 在 `!settings.hooks` 时提前返回，因此该边界需要确认产品预期：如果“完全卸载”也承诺删除 tagged `statusLine`，当前实现可能遗漏这个形态。
