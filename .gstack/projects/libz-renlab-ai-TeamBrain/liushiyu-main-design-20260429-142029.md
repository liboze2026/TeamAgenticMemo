# Design: Recording Memory for Organization Materials

Generated by /office-hours on 2026-04-29
Branch: main
Repo: TeamBrain
Status: APPROVED
Mode: Startup

## Problem Statement

TeamBrain users already have useful organization recordings, transcripts, and AI summaries, but the materials are scattered across Feishu, Google Drive, LAN paths, public links, and local folders. The painful step is not transcription quality. The painful step is that Claude Code cannot reliably discover, index, or cite those materials unless a human manually pastes them into the chat.

The product should turn messy organization materials into structured agent-loadable memory.

In the user's words:

- "大家散落在飞书/Google Drive/本地文件夹，然后有人手工贴给 CC"
- "CC was deaf to learn from mp4 and need to read transcript"
- "it was just too messy and need to re-structure them and make it easy to be indexed by agents. let agents find it out."

## Demand Evidence

The strongest demand evidence is current manual behavior. The user personally uploads or pastes a raw MP4 pointer, software-generated full transcript, and AI summary into Claude Code when the agent needs organizational context. That means the information already exists, the user already trusts the transcript/summary enough to use it, and the bottleneck is repeated manual restructuring.

This is internal demand, not market demand. The immediate user is the TeamBrain builder and later other organization members who want Claude Code to remember prior recordings without manual copy-paste.

## Status Quo

Current workflow:

1. Recording lives in Feishu, Google Drive, LAN path, public URL, or local folder.
2. A separate tool generates the full transcript and an AI summary.
3. The user manually gives Claude Code the MP4 plus transcript plus summary.
4. Claude Code can only use what was pasted in this session.
5. Future agents do not reliably discover the material.

Cost:

- Repeated copy-paste.
- Lost context between sessions.
- No durable "use when" trigger.
- No source chain from answer back to raw file and transcript.
- High chance the user omits the relevant meeting because the material is messy.

## Target User & Narrowest Wedge

Target user: the TeamBrain builder first, then organization members who already record meetings or calls and already have generated transcripts/summaries.

The chosen wedge is C -> B -> A:

1. C: import existing recordings/transcripts/summaries into TeamBrain as structured searchable memory.
2. B: add a manual upload/import command for new materials.
3. A: later add automatic discovery from Feishu, Google Drive, LAN folders, or public folders.

This order matters. If TeamBrain starts with auto-upload, it risks building a clean pipe into an unusable swamp. The first useful product is not "recording upload." It is "agents can find the right transcript when working."

## Constraints

- MVP must not require Claude Code to read or transcribe MP4.
- MVP input is text-first: transcript, optional summary, raw file path, uploader, and use_when.
- Raw file path can be LAN-accessible or publicly accessible.
- Scope is only `public` or `private`. No complex ACL in the first version.
- Automatic loading must be conservative. Inject small summaries and citations first, then expand only when needed.
- All user-visible system actions should go through AttributionBus, not direct console logging.
- New Ports require contract tests before implementation.
- `packages/core/` must remain pure and avoid filesystem/process IO.

## Landscape Findings

Public research shows that "meeting notes + searchable transcript chat" is already table stakes. Otter AI Chat answers questions across meetings, Fireflies AskFred supports natural language questions over one or multiple meeting transcripts, and Granola Chat supports different scopes such as all notes, one meeting, folders, or selected meetings.

Sources:

- Otter AI Chat: https://otter.ai/chat
- Fireflies AskFred docs: https://docs.fireflies.ai/askfred/overview
- Granola Chat docs: https://docs.granola.ai/help-center/getting-more-from-your-notes/chatting-with-your-meetings

The current market direction is broader workspace search. TechCrunch reported on 2026-04-28 that Otter is moving into enterprise search using MCP connections to tools like Gmail, Google Drive, Notion, Jira, and Salesforce.

Source:

- TechCrunch on Otter enterprise search: https://techcrunch.com/2026/04/28/otters-new-feature-lets-users-search-across-their-enterprise-tools/

Enterprise knowledge reports point at the same pain. KMWorld's 2026 State of KM & AI report names AI/automation, access and discovery, and employee productivity as top priorities, while describing information scattered across systems like Slack, Google Drive, and SharePoint.

Source:

- KMWorld 2026 State of KM & AI Report: https://kwfoundation.org/wp-content/uploads/2026/04/2026-State-of-KM-AI-Report.pdf

Transcript retrieval also has a structural problem. Meeting transcripts contain speakers, time, topic drift, and cross-turn context. A 2025 AutoMin paper on RAG over meeting transcripts reports that adding AMR helped produce high-quality responses for about 35% of questions, especially participant-distinguishing questions.

Source:

- GETALP@AutoMin 2025: https://arxiv.org/abs/2508.00476

Implication: TeamBrain should not compete as another meeting assistant. It should specialize in making organization recordings agent-discoverable inside Claude Code and future coding agents.

## Premises

1. The core product is organization material structuring plus agent retrieval, not audio transcription.
2. The MVP should not read MP4 content. It should store the raw file pointer and rely on provided transcript/summary text.
3. Required fields are `raw_file_path`, `transcript`, `uploader`, and `use_when`.
4. Optional fields may include summary, title, participants, created_at, meeting_time, tags, source_tool, checksum, and speaker map.
5. Loading must be conservative: inject short summary and citation first, expand transcript only on explicit query or high-confidence need.
6. Permissions are only `public` and `private` in v1. `public` means team-visible. `private` means only the local user.
7. Recording materials should be a first-class domain object, not forced into `KnowledgeEntry`.

## Approaches Considered

### Approach A: Transcript Import First

Add `teamagent recording import` and store raw file path, transcript, optional summary, uploader, and use_when in a simple recording table. Manual query comes first; automatic loading comes later.

Effort: S
Risk: Low

Pros:

- Fastest way to validate import and search.
- Avoids audio processing and cloud API work.
- Easy to test.

Cons:

- Still feels like a utility script.
- May under-design the object model.
- Future automatic discovery would need refactoring.

Reuses:

- SQLite storage patterns.
- Semantic retriever.
- UserPromptSubmit injection path.
- AttributionBus.

### Approach B: Recording Memory Object

Create a first-class `RecordingMemory` model with import, storage, chunking, semantic indexing, manual query, and conservative agent-context injection. This object owns raw file pointer, transcript, optional summary, uploader, scope, use_when, metadata, chunks, and source references.

Effort: M
Risk: Medium

Pros:

- Clean long-term architecture.
- Supports C -> B -> A naturally.
- Keeps transcript-sized data out of `KnowledgeEntry`.
- Makes source-chain audit possible.
- Allows future Feishu/Drive/LAN adapters without changing the core object.

Cons:

- Requires new Port contracts.
- Requires SQLite schema migration.
- Requires careful chunking and context budget design.

Reuses:

- `packages/ports` contract pattern.
- `packages/adapters/src/storage/sqlite/schema.ts` migration style.
- `SqliteSemanticRetriever` and vector sync patterns.
- `packages/cli/src/bin-user-prompt-submit.ts` additionalContext injection.
- AttributionBus event pattern.

### Approach C: KnowledgeEntry Hack

Store recording summaries and transcript snippets as ordinary `KnowledgeEntry` rows, using trigger/use_when as the trigger and evidence for raw text.

Effort: XS
Risk: High

Pros:

- Fastest demo.
- Minimal schema changes.

Cons:

- Pollutes the rule knowledge model.
- Transcript data is too large and too source-like for `KnowledgeEntry`.
- Makes raw file traceability, chunking, and private/public scope awkward.
- Hard to evolve into automatic discovery.

Reuses:

- Existing `knowledge` table and rule retrieval.

## Recommended Approach

Choose Approach B: Recording Memory Object.

This is the right amount of structure. TeamBrain already learned that forcing unrelated knowledge types into one rule table creates retrieval and calibration problems. Meeting transcripts are not rules. They are source materials with provenance and retrieval affordances.

The recommended product shape:

```ts
type RecordingScope = "public" | "private";

interface RecordingMemory {
  id: string;
  scope: RecordingScope;
  raw_file_path: string;
  transcript: string;
  uploader: string;
  use_when: string;
  summary?: string;
  title?: string;
  tags?: string[];
  source_tool?: string;
  created_at: string;
  updated_at: string;
}

interface RecordingChunk {
  id: string;
  recording_id: string;
  kind: "summary" | "transcript_chunk";
  text: string;
  start_offset?: number;
  end_offset?: number;
  speaker?: string;
}
```

The actual implementation should refine this with Zod schemas in `packages/types`, a Port in `packages/ports`, and SQLite implementation in `packages/adapters`.

## Proposed Architecture

### Domain Layer

Add types:

- `RecordingMemory`
- `RecordingChunk`
- `RecordingImportInput`
- `RecordingQuery`
- `RecordingSearchResult`

Add pure core helpers:

- `normalizeRecordingInput(input)`
- `deriveRecordingSummaryFallback(transcript)` only if no summary is provided and only as a pure formatting fallback, not LLM work.
- `chunkTranscript(transcript, options)`
- `formatRecordingInjection(results, budget)`
- `shouldInjectRecording(userMessage, candidates, sessionSeenIds, budget)`

Core must not touch filesystem or network.

### Port Layer

Add `RecordingMemoryStore` contract before implementation:

```ts
interface RecordingMemoryStore {
  add(recording: RecordingMemory, chunks: RecordingChunk[]): void;
  getById(id: string): RecordingMemory | undefined;
  query(options: RecordingQuery): RecordingSearchResult[];
  list(options: { scope?: "public" | "private"; limit?: number }): RecordingMemory[];
  delete(id: string): boolean;
}
```

Contract tests should cover:

- Add/get roundtrip.
- Required field validation.
- Public/private filtering.
- Query respects scope.
- Query returns source id and raw file path.
- Duplicate id behavior.
- Archived/deleted behavior if supported in v1.

Expose contract suite through `@teamagent/ports/contracts`.

### Storage Layer

Add SQLite tables alongside `knowledge`, not inside it:

```sql
CREATE TABLE recording_memory (
  id TEXT PRIMARY KEY,
  scope TEXT NOT NULL CHECK(scope IN ('public','private')),
  raw_file_path TEXT NOT NULL,
  transcript TEXT NOT NULL,
  summary TEXT DEFAULT '',
  uploader TEXT NOT NULL,
  use_when TEXT NOT NULL,
  title TEXT DEFAULT '',
  tags TEXT DEFAULT '[]',
  source_tool TEXT DEFAULT '',
  status TEXT NOT NULL DEFAULT 'active',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE recording_chunks (
  id TEXT PRIMARY KEY,
  recording_id TEXT NOT NULL,
  kind TEXT NOT NULL CHECK(kind IN ('summary','transcript_chunk')),
  text TEXT NOT NULL,
  start_offset INTEGER,
  end_offset INTEGER,
  speaker TEXT DEFAULT '',
  FOREIGN KEY(recording_id) REFERENCES recording_memory(id) ON DELETE CASCADE
);
```

If sqlite-vec is available, add `recording_chunk_vec` for semantic retrieval. If unavailable, fallback to FTS or keyword search. This should mirror the existing graceful degradation in `schema.ts`.

### CLI Layer

Milestone C command:

```bash
pnpm teamagent recording import \
  --raw-file-path "<path-or-url>" \
  --transcript ./meeting.txt \
  --summary ./summary.md \
  --uploader liushiyu \
  --use-when "When working on TeamAgent recording memory or decisions from this meeting" \
  --scope private
```

Manual query:

```bash
pnpm teamagent recording query "what did we decide about recording memory?"
pnpm teamagent recording list --scope private
pnpm teamagent recording show <id>
```

Later B command can accept `--file` and call a user-configured transcript provider, but not in MVP.

### Claude Code Loading

Extend `UserPromptSubmit` with a recording retrieval block after rule retrieval:

1. Embed or keyword-search the user prompt.
2. Search `recording_chunks` constrained by scope.
3. Exclude already-injected recording ids for this session.
4. Inject only:
   - title or generated label
   - short summary or best chunk
   - raw_file_path
   - uploader
   - use_when
   - command hint to expand full transcript

Example injected context:

```text
◈ TeamAgent relevant recording memory
- Recording: "Recording Memory Brainstorm"
  Why relevant: use_when matches "recording import / transcript indexing"
  Summary: We decided MVP stores raw_file_path + transcript + uploader + use_when...
  Source: /Volumes/team/meetings/2026-04-29-recording-memory.mp4
  Expand: pnpm teamagent recording show rec_123 --transcript
```

Do not inject the full transcript by default.

### Attribution Events

Emit structured events:

- `recording.import.started`
- `recording.import.completed`
- `recording.import.failed`
- `recording.query.completed`
- `recording.injected`
- `recording.expanded`

These should flow through AttributionBus so the user can see what TeamAgent did without direct console logging from core components.

## Milestones

### M-rec-1: RecordingMemory Port + Schema

Deliver:

- Types in `packages/types`.
- `RecordingMemoryStore` Port in `packages/ports`.
- Contract tests in `packages/ports/src/__tests__/recording-memory-store-contract.ts`.
- SQLite tables and adapter implementation.

Exit criteria:

- Contract suite passes for in-memory and SQLite implementations.
- `pnpm test` and `pnpm typecheck` pass.

### M-rec-2: Manual Import + Query

Deliver:

- `teamagent recording import`
- `teamagent recording list`
- `teamagent recording query`
- `teamagent recording show`
- Chunking and indexing.

Exit criteria:

- Import a real transcript + raw file path.
- Query returns source id and raw path.
- Public/private filter works.

### M-rec-3: Conservative UserPromptSubmit Injection

Deliver:

- Recording retriever in UserPromptSubmit.
- Session-level dedupe.
- Context budget cap.
- Attribution events.

Exit criteria:

- Prompt about recording memory injects a short recording hint.
- Repeated prompt in same session does not spam the same recording.
- Full transcript is not injected unless explicitly requested.

### M-rec-4: Manual Upload Wrapper

Deliver:

- CLI accepts raw file path plus transcript path in one command.
- Optional summary generation hook can be configured but is not required.
- Import remains deterministic and testable without network.

Exit criteria:

- User can run one command for a new recording.
- No hidden external service required.

### M-rec-5: Automatic Discovery

Deliver later:

- LAN folder watcher or scan command.
- Feishu/Drive adapters only after manual workflow proves useful.
- Debounce, checksum, duplicate detection.

Exit criteria:

- New transcript appears in watched source and becomes queryable without manual import.

## Open Questions

1. Should `public` recording memories live in project `.teamagent/knowledge.db`, a team-shared repo artifact, or both?
2. Should `private` memories be local-only under the current project, or global across all local projects?
3. Should transcript text be stored directly in SQLite, or should SQLite store pointers to markdown files for large transcripts?
4. What is the maximum transcript size v1 should accept?
5. Should `summary` be optional forever, or should import generate a cheap fallback summary from transcript headings/chunks?
6. Should raw file paths be validated for existence, or accepted as opaque references?

## Success Criteria

- A user can import one existing recording with raw file path, transcript, uploader, and use_when.
- A later Claude Code prompt about a related task receives a short recording memory hint automatically.
- The hint includes enough source information to inspect the original recording or transcript.
- The full transcript is not dumped into context by default.
- Public/private filtering works.
- The feature ships with Port contract tests before adapter implementation.
- `pnpm teamagent skeleton-demo` remains unbroken at milestone end.

## Distribution Plan

This is part of the existing TeamAgent CLI distribution. No new product distribution channel is needed.

Commands should be available through:

- local development: `pnpm teamagent recording ...`
- packaged CLI after normal TeamAgent release flow

CI/CD should reuse the existing package build and verification gates. Feature verification should include:

1. `claudefast -p` running `teamagent recording --help` and producing canonical JSON/help output.
2. `codex exec` running the same help command and hard-matching canonical JSON.
3. Interactive Claude Code test that imports one recording and exports the transcript evidence into PR contents.
